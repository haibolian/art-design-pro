# 前端设计系统与多仓库管理方案（Vue3 + Element Plus）

## 一、问题背景

- 公司存在多个后台管理系统
- 技术栈：Vue3 + Element Plus
- 各项目分属不同部门，不能使用 monorepo
- 当前已有统一模板，但存在以下问题：
  - 模板复制后逐渐分叉
  - UI 风格难以持续统一
  - 升级困难

---

## 二、核心结论

> 不要继续依赖“模板复制”，而应建设“中央设计系统 + 私有 npm 包分发”。

最终架构：

- 多业务仓库（polyrepo）
- 一个中央设计系统仓库
- 通过私有 npm 包共享能力

---

## 三、总体架构

```
业务项目（多个 repo）
        ↓ 依赖
@company/theme
@company/ui
@company/admin-patterns
        ↑ 发布
中央设计系统仓库
```

---

## 四、设计系统拆分

### 1. @company/theme（主题层）

职责：统一视觉体系

内容：
- 颜色（主色/状态色）
- 字号
- 间距
- 圆角
- 阴影
- CSS Variables
- Element Plus 变量覆盖

目标：
> 所有项目“看起来像同一家公司”

---

### 2. @company/ui（组件层）

职责：对 Element Plus 做二次封装

原则：
- 不重写组件
- 基于 Element Plus 封装

示例组件：
- CButton
- CInput
- CSelect
- CDialog
- CPageTable
- CStatusTag

统一内容：
- 默认样式
- 默认行为
- 交互规范
- props 约定

---

### 3. @company/admin-patterns（页面模式层）

这是最关键的一层。

封装后台高频结构：

- 搜索区 + 表格列表
- 表格工具栏
- 批量操作
- 表单页
- 抽屉编辑
- 弹窗表单
- 详情页结构

目标：
> 统一“页面结构 + 交互流程”，而不仅是组件样式

---

### 4. 工程规范包

- @company/eslint-config
- @company/tsconfig
- @company/stylelint-config

作用：
- 保证代码风格一致
- 提升长期可维护性

---

## 六、npm 包设计原则

### 1. 明确导出（exports）

禁止：
```
import X from '@company/ui/src/...'
```

推荐：
```
import { CButton } from '@company/ui'
```

或：
```
import { CButton } from '@company/ui/button'
```

---

### 2. 必须提供类型定义

- props 类型
- emits
- hooks 返回值

作用：
- 提升开发体验
- 提升 AI 理解能力

---

### 3. 插件化接入

```
app.use(CompanyUI)
```

统一：
- 全局组件注册
- Element Plus 配置

---

## 七、业务项目接入方式

### 1. 安装依赖

```
pnpm add @company/theme @company/ui @company/admin-patterns
```

### 2. 入口接入

```
import '@company/theme/index.css'
import CompanyUI from '@company/ui'

app.use(CompanyUI)
```

### 3. 页面开发

```
<CPageTable />
<CSearchPanel />
<CDialogForm />
```

禁止：
- 自己封装基础组件
- 大量覆盖 `.el-*`

---

## 八、发版与升级策略

### 1. 独立版本

- theme@1.x
- ui@2.x
- patterns@1.x

### 2. 语义化版本

- patch：修复
- minor：新增
- major：破坏性变更

### 3. 渐进升级

- 不强制所有项目同步
- 各项目按节奏升级

## 十、推进顺序（务实）

## 十一、常见错误

### ❌ 只维护模板
→ 无法升级

### ❌ fork 多份模板
→ 分叉失控

### ❌ 深度魔改 Element Plus
→ 升级困难

### ❌ 允许随意覆盖样式
→ 统一失效

### ❌ 只做组件不做页面模式
→ 页面仍然不统一

---

## 十二、最终总结

> 你要管理的不是“很多模板项目”，而是：

**一个上游设计系统 + 多个下游业务项目**

并通过以下手段实现统一：

- Design Tokens
- 统一组件库
- 后台页面模式
- 私有 npm 包
- 文档与规范
- AI 可读上下文

这套体系建立后：

- UI 风格可控
- 升级可控
- 多团队协作成本大幅下降
